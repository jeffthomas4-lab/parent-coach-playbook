## Q1. Is the exact-asset smoke proof bound tightly enough to the immutable artifact, or should the proof itself get a detached checksum inside the artifact receipt?

**Answer: Yes / already addressed — verified locally, not yet runtime-proven.**

**Evidence:**
- `CURRENT_STATE.md`, "exact-artifact static asset proof": each build records one content-hashed asset's path, byte length, SHA-256, and Git SHA; post-deploy smoke fetches that exact path and fails unless status/length/hash match.
- `CURRENT_STATE.md`, "self-contained smoke artifact identity": the smoke runner rejects an asset proof without a full lowercase commit SHA and embeds SHA+path+length+hash so a retained receipt ties directly to the immutable build and deployed bytes.
- `CURRENT_STATE.md`, "detached deployment smoke receipt integrity" (latest entry): "both protected deployment jobs now create a detached SHA-256 sidecar for the sanitized post-deploy smoke receipt, verify the sidecar before upload, and retain the receipt and checksum together." This is exactly the control the question asks about — it exists.
- Honestly scoped: "does not claim a workflow ran, a deployment occurred, or a remote receipt exists" — verified locally only.

**Gap:** The checksum sidecar is self-generated by the same job that produces the receipt. It guards against accidental drift/corruption, not against a compromised/misconfigured CI job asserting both a false receipt and a matching checksum. No independent attestation (e.g., GitHub OIDC/Sigstore-style provenance) binds the receipt to the workflow run identity. Defense-in-depth gap, not a defect.

**Recommended action:**
- **LOCAL/REVERSIBLE:** Add a verifier that can also accept/require an independently-issued attestation of the receipt+checksum pair (GitHub Actions artifact attestation via OIDC — no new secret) rather than only a self-generated sidecar. Purely additive.
- **ELAPSED-TIME-EVIDENCE:** The mechanism still needs one real staging and one real production run to become runtime-proven.

## Q3. Are maintenance behaviors task-specific enough to prevent every ungated Aug–Nov write while preserving the year-round legal watch (S4 deletion/opt-out)?

**Answer: Partly — a potential conflict that the staged evidence does not show reconciled.**

**Evidence:**

**Gap:** Two things are both called "the maintenance behavior." The code-level gate is a **blanket** hold on "any D1 write" Aug–Nov with an override that can only tighten. If Vera's proposal-writing path routes through the same "protected site mutation endpoint" the seasonal control blanket-holds, the non-negotiable S4 exception could be silently overridden by the newer freeze — the opposite of task-specific gating. Also, the nine tasks' "report only" behavior may rest on `SKILL.md` prompt text rather than a code gate (only the scheduler + endpoint are code-tested).

**Recommended action:**
- **LOCAL/REVERSIBLE:** Add a narrowly-scoped write-class exception (a distinct `deletion_opt_out_proposal` path not subject to the calendar hold or `PCD_MAINTENANCE_MODE`) plus three tests: ordinary writes held Aug–Nov; the S4 path succeeds inside that window; the override cannot suppress the S4 path. Pure code+test.

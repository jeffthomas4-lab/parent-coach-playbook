That refusal is correct behavior. Nobody weakened it today, and it will keep failing loudly, every morning, until this is fixed.

**Why it matters.** DATA-MAP.md's 30-day deletion and opt-out SLA is unwatched right now. It has been unwatched since at least 2026-07-14. Two verified failure rows in `agent_runs`, both the same error:

| Run | Timestamp | Error |

**P0.1 is not closed.** The task being re-enabled and the guard escalating loudly (it now sets `needs_you = 1` on every failed run) are real improvements over the silent version.

Neither one puts a human eye on the inbox. Only connecting the right Gmail account does that.

**The steps, as far as I could confirm them.** The scheduled-task system this task runs on has no per-task field for picking which Google account it reads mail through. I checked the tool that creates and updates these tasks directly, and the only controls are the prompt text, the schedule, and enabled/disabled. That means the Gmail connection is almost certainly set once, at the app or account level, and shared by every scheduled task that touches Gmail, including DESK's daily football-inbox triage on `jeffthomas@pugetsound.edu`.

That has one direct consequence worth sitting with before you touch anything: if there is only one global Google connection, pointing it at `jeff@coachjeffthomas.com` fixes Vera and breaks DESK, which needs `jeffthomas@pugetsound.edu` specifically and only that inbox.

Decision points, since I could not verify the exact click path with confidence and would rather hand you the fork than a wrong click:

1. Open the Claude/Cowork app's settings and look for Connectors, Connected apps, or Integrations. Find the current Gmail/Google connection and confirm it shows `jeffthomas@pugetsound.edu`. That would match the failure you're seeing.
2. Check whether that settings screen lets you add a **second**, separately named Google connection (some connector systems support several named accounts side by side; others allow only one active connection for the whole app). This is the fact that decides everything else.
3. If it supports multiple named connections: add `jeff@coachjeffthomas.com` as its own connection, then confirm the `pcd-deletion-monitor` task's prompt (readable via the task's `path` in the scheduled-task list) references that account, not a generic "Gmail" that could resolve to either one.
4. If it only supports one connection for the whole app: you have a real choice to make, not a click to make. Options include running the deletion monitor from a separate machine or session that stays logged into the portfolio inbox, or extending the connector setup to support multiple accounts. I don't have enough visibility into the underlying connector architecture to recommend one over the other; that call is yours.
5. Whichever path you take, once `jeff@coachjeffthomas.com` is the account Vera's task actually reads, trigger the task manually once rather than waiting for tomorrow's 7:04 AM run.

**How you verify it worked.** Check the next `agent_runs` row for `pcd-deletion-monitor` (or `vera`, if the deferred rename has happened by then). A passing run reads a message count, not the account-guard error. If the error text is gone and the row shows `status = success` (or a real triage result), the SLA is being watched again.

## 2. Ship the fix for 296 camp pages returning HTTP 500 right now

**What it is.** Verification found a live bug, root-caused it, and fixed it in source: `/camps/[slug].astro` crashes when a listing's `session_end_date` is null, because the date formatter calls `.split('-')` on a value that can be `null`. Four for four on a random sample returned 500. The blast radius, from a direct D1 count, is 296 of 1,554 approved programs, about 19 percent.

This is not a lane's guess. It is a live, reproducible 500 a parent can hit today by clicking a real listing.

**Why it matters.** It is the one thing from this whole session a parent could actually run into right now, and the site's own sitemap crawl can never catch it, because the sitemap query excludes null dates by construction. A clean sitemap crawl (which SEO ran, and it found zero dead URLs) says nothing about this bug.

**The fix.** Already written, in `src/pages/camps/[slug].astro`. The date formatter now degrades gracefully (both dates, one date, or "Dates to be announced") instead of throwing, and the Event JSON-LD omits `startDate`/`endDate` when null instead of emitting invalid markup. It has not been committed or deployed.

Three sibling files (`camps/index.astro`, `camps/[state]/[city]/index.astro`, `camps/[state]/[city]/[sport]/index.astro`) carry the same latent bug but are safe today only because their queries already filter to future dates. Flagged, not fixed, on purpose, so this pass didn't turn into a seventh lane.

**What you do.** This ships through the normal deploy path for this project, nothing special. The commands are at the bottom of this file under Deployment. Deploying also ships everything else this session landed: the Vera CANARY exemption, the canonical/JSON-LD/OG fixes, the worker-cron rewrite, and the roster's run-log wiring, since none of it has shipped yet either.

**How you verify it worked.** After deploy, load these four slugs and confirm 200, not 500, and that each one reads "Dates to be announced" where the dates used to be blank:

- `/camps/pro-football-camp-colorado-spgs-co-5884-profootballcamp/`

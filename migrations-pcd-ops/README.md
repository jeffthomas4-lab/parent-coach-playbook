# PCD operational D1 migrations

This is the canonical migration lineage for the planned `PCD_OPS_DB` binding. It contains trust, support, demand, notification, and future PCD-specific operational state. It must never be applied to the shared Activity Radar `DB` binding or the Forge Command `FORGE_DB` binding.

Current state: an isolated staging database is migrated and rehearsed through `0022`. The production Worker now declares the separate `PCD_OPS_DB` binding, but a 2026-07-17 read-only remote inventory proved that the production database has all migrations `0011` through `0022` pending and no trust-intake tables. The binding alone does not enable any feature. `0015` defines the privacy-request lifecycle substrate. `0016` defines provider-neutral customer identity and tenancy authority. `0017` defines auditable owner claims, disputes, and proposed edits. `0018` adds hashed, one-time invitations/recovery challenges and minimized security events. `0019` adds append-only proposed-edit history. `0020` adds append-only dispute history and recovery lookup support. `0021` adds idempotent cascade execution and expiring export-artifact evidence. `0022` adds a provider-neutral, test-mode commerce ledger. `0023` defines the provider-neutral affiliate product, merchant, recommendation, evidence, health, incident, aggregate revenue, and equipment-intelligence lifecycle. `0024` defines the governed editorial opportunity lifecycle: opportunities, an append-only lifecycle-event log, a reusable evidence-source registry, briefs, claims and claim-source links, relationship mapping, editorial/SEO review records, human approvals with monetization and disclosure classification, and maintenance (refresh/consolidate/retire) proposals. It stores no raw search query, no support-case transcript, and no child or family identifying data -- opportunity rows carry only a bounded, pre-aggregated `signal_summary`. `0025` defines the `events` table: the shared PCD event bus substrate from Foundations section 4, one append-only log of namespaced domain events (`pcd.<domain>.<event>`) that every department's agents write to instead of each inventing its own notification table. It is distinct from `notification_outbox` (`0012`), which is a delivery-attempt/lease queue scoped to one trust-case flow. `0026` defines `proof_inbox`: the raw intake table for the on-site "share your experience" form described in 15-07 Social Proof Page Standard. It holds unmoderated submissions only -- the published `/proof` page reads `src/data/proof.json`, never this table. `scripts/proof.mjs pull` syncs new rows into `src/data/proof-inbox.json` as status `new` for human review (`list`/`approve`/`reject`/`promote`). Migrations `0023`, `0024`, `0025`, and `0026` are committed local design only: none has been applied to staging or production and none activates any API, account, product claim, link, publication, or revenue/content-status mutation. None enables login, owner access, direct directory writes, provider calls, or commerce. Adding a production migration or enabling a feature requires exact-target review and explicit approval.

Until the legacy `migrations/` lineage is retired or split through a separately reviewed repository migration, CI requires migrations `0011` through `0014` to remain byte-identical to their legacy locations. New PCD operational migrations, beginning with `0015`, belong only here and must not be added to the directory-data lineage.

Remaining activation sequence:

1. Approve the database name, account, region/jurisdiction assumptions, retention policy, owners, and recovery class.
2. Maintain the isolated staging rehearsal and exact migration inventory; the production binding already exists, but it has no applied PCD migration.
3. Run Access, failure-isolation, independent backup/retrieval, retention, and customer-journey gates.
4. Obtain exact-target approval for the production migration range, backup identifier, observation window, and any feature flag change.
5. Apply only the approved production migration range, then prove schema and a bounded synthetic workflow before public enablement.
6. Keep `TRUST_INTAKE_ENABLED`, `DEMAND_TELEMETRY_ENABLED`, `EDITORIAL_LIFECYCLE_ENABLED`, and `PROOF_SUBMIT_ENABLED` false until their independent launch gates pass.

## Event bus boundary

`0025_events.sql` is additive and intentionally unapplied. `src/lib/events.ts` writes to it defensively: a missing `PCD_OPS_DB` binding or a missing/unmigrated `events` table is caught and logged, never allowed to break the caller's real work (e.g. publishing a draft). This means the helper is safe to call from production code before `0025` is applied -- every call is a best-effort no-op until the migration lands -- but it also means an application deploy that starts calling `emitEventSafely` is not, by itself, proof that events are being recorded. Confirm rows exist with a read-only `SELECT count(*) FROM events` after `0025` is applied, the same as any other migration in this file.

# Camps Review — 2026-07-16

**Summary:** The camps-quality cron sweep has not run in 68 days (last successful pass: 2026-05-09), root cause found — a required `scheduler_attempts` table was never migrated, so the redeployed scheduler now throws on every invocation before it can even call the sweep endpoint. As a result 604 of 1,554 approved listings (39%) are already past their session end date and still publicly visible, including one that ended in January 2024. Separately, a 2026-07-13 bulk-approval action pushed 931 pending programs straight to "approved" in a single pass with no date filtering — 503 of those were already expired at the moment they were approved. Pending queue is currently empty (0), so there's nothing new to triage this cycle.

**Database note (carried forward from 2026-07-09 review):** `parentcoachdesk.com` reads/writes the shared **`activity-radar`** D1 database (`programs` joined to `organizations`, binding `DB`, database_id `8cc3694a-...`) confirmed directly against `wrangler.production.jsonc`. All numbers below come from that database. The old standalone `parent-coach-playbook` D1 (`camps` table) is a deprecated leftover and was **not** used for this review's conclusions (it was queried first, produced different-looking numbers, and was discarded once the correct binding was confirmed in the repo config).

## 1. Public listings (`programs` where `pcd_status = 'approved'`, n = 1,554)

- **604 approved listings (39%) have `session_end_date` in the past** and are still `record_status = 'active'` — publicly visible on `/camps/` right now. The oldest is *"Jr. Prep Sports Paradise Classic"* (Island Boy Camp, Salem, OR) with a session that ended **2024-01-13** — over two and a half years ago.
- **Root cause, found in code:** `worker-cron/src/index.ts` (redeployed 2026-07-15/16) and the site's own `src/lib/operations-status.ts` both read/write a `scheduler_attempts` table inside the **`forge-command`** D1 database to record every cron attempt durably before firing the actual sweep. No migration anywhere in the repo (`migrations/`, `migrations-activity-radar/`, or elsewhere) creates that table — confirmed by full-repo search and by querying `forge-command` directly (`sqlite_master` shows no `scheduler_attempts` table, only unrelated Forge Command tables like `agents`, `tasks`, `bugs`). The scheduled handler's very first step is an `INSERT` into that missing table, so it throws immediately, every time, before `fireCampsSweep()` ever calls the production sweep endpoint. This explains why `url_last_checked_at` across all 1,554 approved rows is frozen at 2026-05-09 (a ~1.5-hour window, 20:38–22:18 UTC) — no health check or stale-archive pass has completed since.
- **Editorial-quality incident (2026-07-13):** A single batch, `reviewed_by = 'sonnet-bulk-approval'`, `reviewed_at = 2026-07-13T16:09:49Z`, approved 931 programs in one shot (plus rejected 6). Of those 931, **503 (54%) already had a past `session_end_date` at the moment of approval** — including the 2024-01-13 one above. This looks like the entire backlog the 2026-07-09 review recommended triaging was instead dumped wholesale into "approved" without a date check, rather than reviewed per-item. Worth confirming with whoever ran this whether it was intentional, and whether that process should gate on date before approving in bulk going forward.
- **Duplicates:** 0 exact-duplicate groups remain among approved listings (name + city/state + session dates) — the 2026-07-10 de-dup cleanup has held.
- **Missing required fields:** 0 approved listings are missing `website_url`. Clean.
- **Source concentration:** 1,013 of 1,554 approved listings (65%) are still `ussportscamps.com` (Nike-branded camps). No other domain exceeds 2%. **`domain_quality` has zero rows in this database** — not low counts, an empty table — so `/admin/source-quality` cannot currently render an approval rate for any domain, including this one. This has been flagged in the prior review too and is still unaddressed.

Given the tool restriction, treat this section as a partial check rather than a full 10-site verification. Recommend re-running with browser access (Claude in Chrome or similar) if a real external-link check is wanted this week.
